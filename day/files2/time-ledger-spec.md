# 시간의 궤적 (Time Ledger) — 작업 명세서

**버전** v1.0
**최종수정** 2026-04-23
**작성자** 몽브 × Claude
**프로젝트 유형** 개인 업무 관리 웹 애플리케이션 (React SPA, 로컬 영속 저장)

---

## 1. 프로젝트 개요

### 1.1 배경
운영자는 2개 이커머스 브랜드(아이노즈, 몽쿨)와 5개 사업 엔티티를 동시에 관리하는 이커머스 팀장 역할을 수행한다. 업무 영역이 인증·법무·상세기획·마케팅·CS·채용·운영 등 10개 이상으로 분산되어 있어, 기존 노션 기반 관리는 **입력 장벽이 높고 월/연 단위 패턴 인식이 어려운** 구조적 한계를 드러냈다.

### 1.2 문제 정의
1. **입력 마찰** — 노션은 페이지/데이터베이스 구조 생성에 시간이 소요되어 "기록하는 행위" 자체가 부담이 됨
2. **조망 부재** — 어떤 업무에 시간을 얼마나 썼는지, 어느 날이 유난히 바빴는지 한눈에 보이지 않음
3. **연간 전략 단절** — 연초에 세운 연간 목표/분기 테마가 일상 업무 기록과 연결되지 않음
4. **블라인드 스팟(맹점)** — 인증 갱신일, 정산 이슈, 반복되는 CS 패턴 등 "놓치기 쉬운 항목"이 체계적으로 관리되지 않아 사후에 발견됨

### 1.3 목적
운영자의 시간 사용을 **저마찰 기록 → 월간 패턴 인식 → 연간 전략 정렬 → 블라인드 스팟 사전 방지**의 네 층위로 통합 관리하는 단일 웹 도구 구축.

### 1.4 사용자 페르소나
- 다중 브랜드 운영 이커머스 팀장 1인
- 기술 이해도: 중상 (Claude Code, HTML+localStorage 도구 자체 제작 경험)
- 기기: 데스크톱 주 사용, 모바일 확인 빈도 보통
- 핵심 니즈: "노션보다 가볍고, 엑셀보다 시각적이며, 캘린더보다 분석적인 도구"

## 2. 설계 철학

### 2.1 핵심 원칙
1. **3초 입력 원칙** — 모든 기록은 클릭 3회, 3초 이내에 완료되어야 한다
2. **구조 강요 최소화** — 사용자가 "어디에 입력할지"를 고민하게 만들지 않는다
3. **시각 우위** — 숫자보다 패턴, 표보다 히트맵
4. **층위별 시간 스케일** — 일간(Today) / 월간(Monthly) / 연간(Yearly) / 블라인드스팟(Blind Spots, 시간 무관)
5. **데이터 소유권** — 모든 데이터는 로컬 영속 저장소에 보관되며 사용자가 언제든 내보낼 수 있다

### 2.2 디자인 방향
- **Editorial Minimalism** — 따뜻한 뉴트럴(#FAF8F3 베이스) + 절제된 카테고리 컬러 포인트
- **타이포그래피** — 국문 Pretendard Variable + 영문/숫자 Fraunces (italic display)
- **여백 우선** — 정보 밀도보다 조망 품질 우선
- **AI slop 회피** — 보라 그라디언트, Inter/Roboto, 대시보드 클리셰 배격

## 3. 기능 명세

### 3.1 일간 (Today) — [구현 완료]

#### 기능 요약
하루 단위 업무를 "계획 → 기록 → 달성률" 흐름으로 관리

#### 상세 명세
- **날짜 네비게이션**: 좌우 화살표로 과거/미래 일자 이동, 오늘 표시 강조
- **상단 요약 카드 3종**: 계획 시간, 실행 시간, 달성률 (색상 게이지 포함)
- **오늘의 설계 (Plan Section)**
  - 카테고리 + 예상 시간 + 메모 입력
  - 추가/삭제 가능, 카테고리별 색상 좌측 바 표시
- **오늘의 궤적 (Log Section)**
  - 카테고리 + 내용(짧은 텍스트) + 실제 시간 입력
  - Enter 키 지원, 시간순 자동 정렬
  - 입력 즉시 색상 도트 + 타임스탬프 표시

#### 수용 조건
- 빈 날짜 진입 시 안내 플레이스홀더 표시
- 과거 날짜 수정 시 즉시 반영, 월간 뷰에 연동

---

### 3.2 월간 (Monthly) — [구현 완료]

#### 기능 요약
한 달 업무량을 GitHub 스타일 히트맵과 카테고리 분포 차트로 조망

#### 상세 명세
- **월 네비게이션**: 좌우 화살표로 전후월 이동
- **월간 요약 카드**: 총 기록 시간, 활성 일수, 일평균
- **카테고리 필터**: 전체 또는 특정 카테고리 1개 선택
  - 필터 적용 시 히트맵 색상이 해당 카테고리 색으로 변경
- **히트맵**
  - 7열 그리드 (일~토, 주말 색상 구분)
  - 5단계 색상 농도 (0, 0-25%, 25-50%, 50-75%, 75-100%)
  - 셀 호버 시 툴팁, 클릭 시 해당 날짜로 이동
- **카테고리별 분포 파이차트**
  - 시간 기준 상위 정렬, 카테고리별 색상/시간/비율 리스트 병기

#### 수용 조건
- 기록이 없는 월은 "이 달에는 아직 기록이 없습니다" 플레이스홀더
- 오늘 날짜는 히트맵에서 outline으로 강조

---

### 3.3 연간 (Yearly) — [신규 · v1.1 개발 예정]

#### 기능 요약
연 단위 전략(목표/분기 테마/핵심 프로젝트)을 설정하고, 월별 실제 기록과 대조하여 전략 정렬도를 시각화

#### 상세 명세

**A. 연간 목표 (Yearly Goals)**
- 연간 핵심 목표 3~5개 등록 (OKR 스타일)
- 각 목표: 제목, 설명, 관련 카테고리(복수), 진행률(수동 0~100%)
- 목표 상태: 진행중 / 달성 / 보류 / 철회

**B. 분기 테마 (Quarterly Themes)**
- Q1~Q4 각각 한 줄 테마 문장 입력
- 예: Q1 "인증·법무 정비", Q2 "신상품 런칭 집중", Q3 "해외 확장 검토", Q4 "성수기 대응"
- 테마별 핵심 프로젝트 3~5개 리스트 (체크리스트 형식)

**C. 연간 히트맵**
- 12개월 × 31일 매트릭스 (일별 총 시간 색상 농도)
- 월별 집계 사이드바: 총 시간, 상위 카테고리 3종
- 클릭 시 해당 월간 뷰로 이동

**D. 카테고리별 연간 트렌드 라인차트**
- X축 월, Y축 시간
- 카테고리 복수 선택 가능 (라인 색상 = 카테고리 색)
- "계획했던 비중 vs 실제 비중" 비교 옵션 (분기 테마 기반)

**E. 월별 성과 카드**
- 12개 월 카드 형태로 나열
- 각 카드: 총 시간, 활성일수, 핵심 이벤트 1줄 메모, 상위 카테고리 3종

#### 수용 조건
- 연도 전환 네비게이션 제공 (과거 연도 조회 가능)
- 데이터가 없는 월도 카드는 표시되되 플레이스홀더 처리

### 3.4 블라인드 스팟 (Blind Spots) — [신규 · v1.1 개발 예정]

#### 기능 요약
이커머스 운영에서 놓치기 쉬운 항목을 사전 등록·관리하고, 사후 발견한 누락 사항도 아카이빙하여 재발 방지

#### 개념 정의
"블라인드 스팟"은 다음 세 유형을 포괄한다:
1. **데드라인형 (Deadline)** — 특정 일자에 반드시 처리해야 하는 항목 (인증 갱신, 상표 갱신, 계약 만료 등)
2. **주기형 (Recurring)** — 정기적으로 점검·실행해야 하는 항목 (월간 정산 확인, 분기 재고 감사, 광고 소재 교체 등)
3. **이슈 로그형 (Incident)** — 사후에 "아 이거 놓쳤네"를 기록하여 패턴 학습 및 재발 방지 (CS 패턴, 경쟁사 동향 누락, 시즌 대응 지연 등)

#### 상세 명세

**A. 등록 항목 구조**
```
- id (auto)
- title (string, 필수)
- type (enum: deadline / recurring / incident)
- category (enum: 인증·법무 / 정산·재무 / 재고·공급망 / 마케팅 / 계약 / 고객경험 / 플랫폼정책 / 기타)
- dueDate (date, type=deadline일 때 필수)
- recurrence (object, type=recurring일 때: {frequency: weekly|monthly|quarterly|yearly, anchor: date})
- description (string, 상세 메모)
- severity (enum: low / medium / high / critical)
- status (enum: pending / in-progress / done / missed / dismissed)
- occurredDate (date, type=incident일 때 "놓친 시점" 기록)
- lessonLearned (string, type=incident일 때 "다음엔 이렇게 방지" 메모)
- relatedCategoryIds (string[], Time Ledger 카테고리와 연결)
- createdAt, updatedAt (datetime)
```

**B. 대시보드 뷰 (기본)**

섹션 4개 탭 또는 세로 스크롤:
1. **임박 (Upcoming)** — 7일 이내 도래하는 데드라인 및 주기형 항목
2. **대기 (Pending)** — 모든 미해결 항목, 날짜순 정렬
3. **놓친 이슈 (Incident Archive)** — 사후 기록된 이슈 로그, 카테고리별 필터 및 검색
4. **완료 (Done)** — 최근 완료 항목 (월별 롤오버)

**C. 블라인드 스팟 캘린더 뷰**
- 월간 캘린더 형태, 데드라인 및 주기형 항목이 해당 일자에 표시
- Time Ledger의 월간 히트맵과 별도 레이어로 오버레이 옵션 제공 (토글)

**D. 알림 및 임박 표시**
- 브라우저 내 시각적 알림만 우선 (v1.1) — 상단 배지, 임박 항목 붉은색 강조
- 이메일/푸시 알림은 v1.2 이후 검토

**E. 기본 템플릿 제공**
초기 진입 시 이커머스 운영자용 블라인드 스팟 템플릿 자동 로드 옵션:

| 항목 | 유형 | 주기/일자 | 카테고리 |
|---|---|---|---|
| KC 인증 갱신 확인 | 주기 | 연간 | 인증·법무 |
| 특허 연차료 납부 | 주기 | 연간 | 인증·법무 |
| 상표 갱신 확인 | 데드라인 | 개별 | 인증·법무 |
| 플랫폼별 정산 내역 대조 | 주기 | 월간 | 정산·재무 |
| 부가세 신고 | 주기 | 분기 | 정산·재무 |
| 재고 소진 예상일 점검 | 주기 | 주간 | 재고·공급망 |
| 광고 소재 교체 | 주기 | 월간 | 마케팅 |
| 체험단 리뷰 업로드 마감 | 데드라인 | 개별 | 마케팅 |
| 공급사 계약 만료 | 데드라인 | 개별 | 계약 |
| CS 반복 문의 FAQ 반영 | 주기 | 월간 | 고객경험 |
| 네이버·쿠팡 정책 공지 확인 | 주기 | 주간 | 플랫폼정책 |
| 경쟁사 주요 변동 모니터링 | 주기 | 월간 | 기타 |

(사용자가 필요 항목만 체크하여 추가 가능)

#### 수용 조건
- 주기형 항목은 완료 시 다음 주기로 자동 롤오버
- 데드라인 항목은 완료 시 아카이브, 미완료 + 날짜 경과 시 status=missed 자동 전환
- 이슈 로그는 카테고리별 빈도 집계 제공 ("고객경험 이슈가 3개월간 5건 발생")

## 4. 정보 구조 및 데이터 모델

### 4.1 저장소
- **저장 방식**: `window.storage` API (영속 키-값 저장소, localStorage 대체)
- **키 네이밍**:
  - `categories` — Time Ledger 업무 카테고리 배열
  - `entries` — 일자별 기록 객체 (전체)
  - `yearly:{YYYY}` — 연도별 목표/테마 객체
  - `blindspots` — 블라인드 스팟 전체 배열
  - `blindspot_categories` — 블라인드 스팟 카테고리 배열
  - `settings` — 앱 설정 객체

### 4.2 주요 스키마

```typescript
type Category = {
  id: string;
  name: string;
  color: string; // hex
};

type LogEntry = {
  id: string;
  categoryId: string;
  content: string;
  hours: number;
  timestamp: string; // ISO
};

type PlanItem = {
  categoryId: string;
  hours: number;
  note: string;
};

type DayEntry = {
  plan: PlanItem[];
  logs: LogEntry[];
};

type Entries = Record<string, DayEntry>; // key: YYYY-MM-DD

type YearlyPlan = {
  year: number;
  goals: {
    id: string;
    title: string;
    description: string;
    relatedCategoryIds: string[];
    progress: number; // 0-100
    status: 'active' | 'done' | 'paused' | 'dropped';
  }[];
  quarters: {
    q: 1 | 2 | 3 | 4;
    theme: string;
    projects: { id: string; title: string; done: boolean }[];
  }[];
  monthNotes: Record<number, string>; // 1-12
};

type BlindSpot = {
  id: string;
  title: string;
  type: 'deadline' | 'recurring' | 'incident';
  category: string;
  dueDate?: string;
  recurrence?: {
    frequency: 'weekly' | 'monthly' | 'quarterly' | 'yearly';
    anchor: string;
  };
  description: string;
  severity: 'low' | 'medium' | 'high' | 'critical';
  status: 'pending' | 'in-progress' | 'done' | 'missed' | 'dismissed';
  occurredDate?: string;
  lessonLearned?: string;
  relatedCategoryIds: string[];
  createdAt: string;
  updatedAt: string;
};
```

## 5. UI/UX 원칙

### 5.1 내비게이션 구조
```
Header (브랜드 + 오늘 날짜)
└─ TabBar: [오늘] [월간] [연간] [블라인드 스팟] [설정]
   └─ Main (max-width: 64rem, 중앙 정렬)
```

### 5.2 인터랙션 규칙
- 모든 입력 필드에 Enter 키 제출 지원
- 파괴적 액션(삭제, 초기화)은 `confirm` 다이얼로그 필수
- 호버 시에만 삭제 아이콘 노출 (미관 + 실수 방지)
- 색상은 오직 카테고리 식별 용도, 기능적 컬러(성공/실패)는 지정 팔레트에서만 사용

### 5.3 반응형
- 데스크톱 우선 설계 (1024px+)

## 8. 비기능 요구사항

- **성능**: 5년치 기록(약 1,800일) 로딩 시 초기 렌더 1초 이내
- **데이터 안전성**: 저장 실패 시 사용자 알림, 세션 내 메모리 상태 보존
- **접근성**: 키보드만으로 모든 입력 가능, 색맹 고려 패턴 병기(미래)
- **백업**: JSON 내보내기를 통한 사용자 자가 백업 권장 주기(월 1회) 안내

## 10. 부록: 구현 시 참고 사항

### 10.1 영속 저장소 API 래퍼 예시
```javascript
const storage = {
  async load(key, fallback) {
    try {
      const r = await window.storage.get(key);
      return r?.value ? JSON.parse(r.value) : fallback;
    } catch { return fallback; }
  },
  async save(key, value) {
    try { await window.storage.set(key, JSON.stringify(value)); return true; }
    catch (e) { console.error(e); return false; }
  }
};
```

### 10.2 주기형 블라인드 스팟 롤오버 로직
완료(`status=done`) 시:
1. 현재 레코드는 status 유지하되 archive 플래그 추가
2. 새 레코드 생성: `dueDate = 현재 완료일 + recurrence.frequency`
3. 신규 레코드는 `status=pending`으로 진입

### 10.3 연간 히트맵 최적화
- 12 × 31 = 372 셀, 초기 로드 시 해당 연도의 `entries` 필터링 1회 수행
- `useMemo`로 연도 변경 시에만 재계산
- 날짜 존재 여부는 `new Date(year, month, 0).getDate()`로 판별
